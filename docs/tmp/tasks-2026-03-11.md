# 2026-03-11 リファクタリングタスク一覧

## この文書の目的

この文書は、[docs/CURRENT_ARCHITECTURE.md](../CURRENT_ARCHITECTURE.md) と [docs/ARCHITECTURE.md](../ARCHITECTURE.md) を比較し、今後行うべき責務委譲・ファイル移動・不要物整理をまとめるための実務メモである。

ここでは「今すぐ全部やる」ことではなく、**どこをどう整理すると目標構造へ近づくか**を明示する。

## 比較結果の要約

大きな差分は次の 6 点である。

1. `desktopApp` に集まる責務が多すぎる
2. `canvas` と呼べる独立層がまだない
3. `ui-shell` が panel runtime と panel presentation を兼務している
4. project / workspace I/O が plugin 主導ではなく host 主導である
5. ツール実行が plugin 中心ではなく host 内蔵実装に依存している
6. 命名と実態がずれている crate / module が残っている

---

## 優先度A: まず責務を切り離す

### A-1. `DesktopApp` を薄くする

**現状**

- `apps/desktop/src/app/mod.rs`
- `apps/desktop/src/app/commands.rs`
- `apps/desktop/src/app/state.rs`
- `apps/desktop/src/app/present.rs`
- `apps/desktop/src/app/input.rs`

が、状態・副作用・保存・UI同期・描画計画を同時に持っている。

**目標**

`desktopApp` は次だけに絞る。

- 起動
- ランタイム I/O
- event loop
- OS 入力の収集
- 各専門層の呼び出し順制御

**委譲先**

- canvas 処理 → 新設 `canvas`
- 表示計画 → `render`
- plugin / panel API 仲介 → `ui-shell`
- 計測 / native dialog / desktop config → `desktop-support`
- project / workspace 操作の上位機能 → plugin

**着手候補**

1. `DesktopApp` から I/O orchestration と domain update を分ける
2. `present.rs` から render plan 相当を `render` へ寄せる
3. `commands.rs` から plugin 化対象の保存/読込操作を切り離す

### A-2. `canvas` 層を新設する

**現状**

canvas 処理は次へ分散している。

- `crates/app-core/src/document.rs`
- `crates/app-core/src/painting.rs`
- `apps/desktop/src/canvas_bridge.rs`
- `apps/desktop/src/app/drawing.rs`
- `apps/desktop/src/app/input.rs`
- `crates/render/src/lib.rs`

**目標**

`canvas` は次を担う。

- キャンバス入力の解釈
- ツール実行のランタイム
- 差分生成と適用
- ブラシ/消しゴム/塗りつぶし等の canvas オペレーション
- `app-core` と `render` の間にある canvas 専用ロジック

**ファイル移動候補**

- `apps/desktop/src/app/drawing.rs`
  - → `crates/canvas/src/runtime/` もしくは `crates/canvas/src/tools/`
- `apps/desktop/src/canvas_bridge.rs`
  - → `crates/canvas/src/input_bridge.rs`
- `crates/app-core/src/painting.rs`
  - → `crates/canvas-api/` か `crates/canvas/` へ再配置検討
- `crates/app-core/src/document.rs` 内の paint context 解決まわり
  - → `canvas` へ抽出

**注意**

- `Document` に残すのは canvas state と command 適用に必要な純粋状態だけに寄せる
- ツール実行自体は `canvas` へ移す

### A-3. `ui-shell` を runtime と presentation に分割する

**現状**

`crates/ui-shell` が次を同時に持つ。

- panel discovery
- DSL 読込
- Wasm 実行
- host snapshot 同期
- local state / persistent config
- layout / hit-test / focus / text input
- panel surface 構築

**目標**

- runtime 側
  - plugin discovery
  - DSL / Wasm bridge
  - state patch / command mapping
  - plugin config / host snapshot 同期
- presentation 側
  - layout
  - hit-test
  - focus
  - text input
  - panel surface 生成

**分割候補**

runtime 側へ寄せる候補:

- `crates/ui-shell/src/dsl.rs`
- `crates/ui-shell/src/workspace.rs` の一部
- `crates/ui-shell/src/lib.rs` 内の panel registry / config / sync ロジック

presentation 側に残す候補:

- `crates/ui-shell/src/presentation.rs`
- `crates/ui-shell/src/surface_render.rs`
- `crates/ui-shell/src/focus.rs`
- `crates/ui-shell/src/tree_query.rs`

**将来候補名**

- `crates/panel-runtime`
- `crates/ui-shell`

## 優先度B: plugin 主導へ寄せる

### B-1. project / workspace I/O を plugin 主導へ移す

**現状**

- project save/load は `storage`
- session / preset / template は `desktop-support`
- その orchestration は `DesktopApp`
- UI は `plugins/app-actions` と `plugins/workspace-presets` が担うが、実処理は host 依存が強い

**目標**

plugin が次を主導する。

- project 読込/保存
- workspace 読込/保存
- workspace panel 配置管理

**必要な委譲**

- host は file dialog / path / serializer / runtime document access を service として公開する
- plugin はその service を呼んで I/O を構成する

**整理対象ファイル**

- `apps/desktop/src/app/commands.rs`
- `crates/storage/src/project_file.rs`
- `crates/storage/src/project_sqlite.rs`
- `crates/desktop-support/src/session.rs`
- `crates/desktop-support/src/workspace_presets.rs`
- `plugins/app-actions/src/lib.rs`
- `plugins/workspace-presets/src/lib.rs`

**方針**

- 低レベル serializer はすぐ消さず内部実装として残してよい
- ただし「何を保存するか」「いつ保存するか」の主導権は plugin 側へ寄せる

### B-2. ツールシステムを plugin 中心へ再設計する

**現状**

- `storage::tool_catalog` が `tools/` を読む
- `Document` が tool 定義と設定を保持する
- 実際の描画差分生成は `apps/desktop/src/app/drawing.rs` の built-in 実装が中心

**目標**

plugin が次を担う。

- ツール一覧の読込 / 一覧表示
- ツールパラメータの読込 / 一覧表示 / 設定
- 親ツールと子ツールの読み込み
- 入力座標から canvas 差分を生成する処理

**移行タスク**

1. `ToolDefinition` を「plugin を指す定義」と「runtime 実行状態」に分ける
2. built-in paint 実装を `apps/desktop` から切り出す
3. ツール plugin が parameter file と child tool を読める API を用意する
4. host は差分の適用ランタイムだけを担う

**整理対象ファイル**

- `crates/storage/src/tool_catalog.rs`
- `crates/app-core/src/document.rs`
- `apps/desktop/src/app/drawing.rs`
- `plugins/tool-palette/src/lib.rs`
- `plugins/pen-settings/src/lib.rs`

### B-3. view / panel list / color palette を plugin-first に揃える

**現状**

- `plugins/view-controls`
- `plugins/panel-list`
- `plugins/color-palette`

は plugin 化されているが、host 側 command mapping と state shape への依存がまだ強い。

**目標**

- plugin は操作を宣言する
- host は安定 API で受け取る
- plugin ごとの個別分岐を減らす

**整理対象**

- `crates/plugin-api/src/lib.rs`
- `crates/panel-sdk/src/commands.rs`
- `crates/ui-shell/src/dsl.rs`
- `apps/desktop/src/app/commands.rs`

## 優先度C: 命名と物理配置を揃える

### C-1. `plugin-api` の役割を再定義する

**現状**

`plugin-api` は実質 panel API である。

**選択肢**

1. 真に汎用 plugin API へ拡張する
2. `panel-api` 相当に改名し、汎用 plugin API は別に作る

**推奨**

- panel 契約は `panel-api` 相当へ寄せる
- 汎用 plugin service 契約は別境界へ切る

### C-2. `panel-sdk` を `plugin-sdk` 体系へ再編する

**現状**

- `panel-sdk`
- `panel-macros`

が panel 専用 authoring surface になっている。

**目標**

- plugin 作者向け入口を `plugin-sdk` として統一する
- panel 関連はそのサブモジュールに置く
- macro はその配下の authoring surface として扱う

### C-3. `render` と desktop 固有 frame 処理の境界を揃える

**現状**

- `render` は scene / transform / panel rasterize を持つ
- `apps/desktop/src/frame/` と `apps/desktop/src/app/present.rs` に host 固有の合成計画が残る

**目標**

- 画面生成ロジックは `render`
- OS/GPU 所有と event loop は `desktopApp`

**移動候補**

- `apps/desktop/src/app/present.rs` の render plan 部分
- `apps/desktop/src/frame/` のうち desktop 固有ではない表示計算

## 実行順の提案

### ステップ1

- `DesktopApp` の責務棚卸し
- `ui-shell` の runtime / presentation 分離設計
- `canvas` 新設の境界定義

### ステップ2

- built-in paint runtime を `apps/desktop` から外へ出す
- tool runtime と `Document` の責務を分離する
- render plan を `render` へ寄せる

### ステップ3

- project / workspace I/O を plugin service 経由へ寄せる
- view / panel list / color / workspace plugin 群の command API を安定化する

### ステップ4

- 命名整理
- 不要サンプル整理
- 旧責務の削除

## この文書の結論

最優先は新機能追加ではなく、

- `DesktopApp` を薄くする
- `canvas` を独立させる
- `ui-shell` を二分割する
- 非性能領域を plugin 主導へ寄せる

の 4 点である。

これが終わると、以後の機能追加は「どこへ置くべきか」で迷いにくくなる。

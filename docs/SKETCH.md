## CSPのどうにかしてくれポイントリスト (随時更新)

- アルゴリズムが悪い
  - ガウシアンフィルタを2次元で素朴にかけている
  - テキストの何かのチェック(O(n)?)をテキストが1文字変更されるごとに回すので長いテキストを書いているとどんどん重たくなるやつ

  - PDF書き出しを1枚ずつ順番に……
  - ほか変形中などにUIも止まるやつやめて

- サブスク (これは営業の問題か……)

- 漫画制作向けの機能をもっと中心に据えたいから
- 既存OSSへの部分改善より、ワークフロー全体を一貫した思想で設計したいから
- 自分でゼロから設計した方が、道具としても学習対象としてもおもしろいから

## この文書の目的

この文書は、アイデアメモではなく以下を定めるための草案とする。

- 誰の、どの作業を、どの程度速く楽にするのか
- MVPで必須の機能は何か
- 実装判断に必要な非機能要件は何か
- 技術スタックをどう選ぶか

実装順序やフェーズ分けは `docs/ROADMAP.md` で扱い、この文書では「何を目指すか」と「何を重要とみなすか」に集中する。

## 目指す体験

目標は「高速」「低負荷」「低認知負荷」で漫画制作を進められること。

- 高速: 起動、描画、保存、書き出し、UI操作の待ち時間が短い
- 低負荷: 大きい原稿や多ページ作品でもメモリ消費とCPU負荷が破綻しない
- 低認知負荷: 画面上に一度に表示されている情報を減らし、今やるべき操作が自然に見える

## 想定ユーザー

まずは以下を主対象にする。

- 個人で漫画・同人誌・ネームを制作する人
- レイヤーやブラシに慣れているが、既存ソフトのUIや思想に疲れている人
- テキストからネームを起こし、コマ単位で作業を進めたい人

将来的には以下も視野に入れる。

- アシスタント込みの分業
- スクリプトやプラグインを自作したい上級者
- 大量ページを扱う商業向けワークフロー

## コアコンセプト

### 1. ページ中心ではなく「コマ中心」で作業できる

- ネーム以降の主要工程では、ページ全体よりコマを主要単位として扱えるようにする
- ただし最終的な入稿や見開き確認はページ/冊子単位で確認できるようにする

### 2. テキストから絵作りへ自然につながる

- セリフや地の文を先に流し込み、コマ配置・文字量・視線誘導を先に確認できるようにする

### 3. ツール配置・制作プロファイルを即座に切り替えられる

- ネーム用、ラフ用、線画用、仕上げ用など、工程に応じたUIプロファイルを切り替えられる
- ユーザー間で設定配布・読み込みができる

### 4. 重い処理でもUIを止めない

- フィルタ、変形、書き出し、サムネイル生成などは非同期・並列で処理し、メインUIをブロックしない

## 必要な特徴

### 編集体験

- キャンバスがある
- キャンバス左右の余白に前後ページまたは前後コマのプレビューを表示できる
- コマの並び換えができる
- コマのスナップが賢い
- コマから飛び出すオブジェクトを簡単に描画できる
  - 例: コマ配下に存在するがコママスクを受けない描画グループを作れる
- セリフをテキストとして流し込みやすい
- レイヤーの木構造をもっと見やすく表示できる
  - たとえばインデント、ズーム、ミニマップ、役割別表示など
  - 同じ機能を持つレイヤーでも、最初から「線画用」のボタンを押してその名前で挿入できるようにする

### テンプレート・UIカスタマイズ

- 組み込みテンプレートとユーザー製テンプレートを同じリストで扱える
- テンプレートを並び換え・お気に入り化・タグ付け・検索できる
- ツールパレットを並び換えできる
- 制作プロファイルを保存・切り替え・共有できる

### 性能・アーキテクチャ

- 変形機能やフィルタ機能が省メモリである
- PDF書き出しなどがマルチスレッドで速い
- 重たい処理中にも画面が停止しない
- 将来的にGPU処理へ拡張しやすい

### 互換性・拡張性

- クリスタのブラシを可能な範囲で読める
- プラグイン開発が容易
- スナップショットを保存可能でブランチを切ることができる

## 現状メモにある曖昧点・追加で決めるべきこと

以下は実装前にもう少し定義した方がよい。

### 1. 「ページ」と「コマ」のどちらを主軸にするか

現状では「少なくともペン入れ段階からはコマが基本単位」とあるが、これはかなり重要な設計分岐になる。

候補:

- A. データ構造の主軸をページにして、コマ編集ビューを強くする
- B. データ構造の主軸をコマにして、ページはレイアウト結果として扱う

当面の提案:

- 内部モデルは「作品 → ページ → コマ → レイヤー」を基本にする
- UI/操作フローでは「コマ中心ビュー」を前面に出す

理由:

- 入稿、見開き、ノンブル、印刷サイズ、PDF出力はページ概念が依然として重要
- 一方で作業効率はコマ単位ビューで大きく改善できる

#### ユーザーからの回答

当面の提案に賛成。

### 2. 「賢いスナップ」の定義

賢いコマ枠スナップは

- ページ枠
- 断ち切り線
- ノド/小口基準
- コマ枠
- 隣接コマの辺や余白
- セーフエリア

などに対して、CADのように2要素間の数値を指定したりしてスナップできる。

- 視覚フィードバック

については未定。

#### 回答

MVPでは、スナップ中の対象線・距離・拘束方向をキャンバス上に簡易表示する。
詳細な視覚フィードバックや数値入力UIは後期実装とする。

### 3. 「クリスタのブラシが読める」の範囲

これは法務面・互換性面・工数面で非常に重い。

要分解:

- どのブラシ形式を対象にするか
- どこまで互換再現するか
  - 形状
  - 散布
  - 筆圧
  - テクスチャ
  - 混色
- 読み込みのみか、編集・再保存も行うか

当面の提案:

- MVPでは「独自ブラシ形式 + 一部の外部ブラシ設定のインポート」に限定する
- 完全互換は後回しにする

#### 回答

当面の提案に賛成。
読み込みコンバータはプラグインとして実装することを予定。

### 4. 「プラグイン開発が容易」の定義

最低限決めたい:

- UI拡張を許すか
- フィルタや入出力だけか
- スクリプト実行言語は何か
- 権限制御はどうするか
- ABI安定性をどう扱うか

当面の提案:

- 初期はWasmtimeによるWASMプラグイン方式を第一候補とする
- 範囲はフィルタ、入出力、テンプレート生成、バッチ処理から始める

#### 回答

当面の提案に賛成。
パネル型のGUIを想定しているので、将来的には新パネルの追加も許容したい。
既存のパネルにパッチを当てる形のプラグインも許容したい。

ただし、拡張の自由度は段階的に開放する。

- MVPでは、新パネル追加・フィルタ・入出力・テンプレート生成・バッチ処理までを許容する
- 既存パネルへの拡張は、明示的なextension pointを用意した箇所のみに限定する
- 既存パネル全体を差し替える仕組みや、UIツリーへの自由なパッチ適用は後期実装とする

また、標準パネルも将来的にプラグイン化できる構造を目指すが、MVPではまず本体モジュールとして実装し、APIの境界が固まってから順次分離する。

### 5. スナップショットとブランチの粒度

決めたいこと:

- 作品単位か、ページ単位か、コマ単位か
- 差分保存かフル保存か
- 画像データの管理方式
- 競合解決やマージをどこまでやるか

当面の提案:

- まずはローカル専用の軽量スナップショット機能から始める
- 「ブランチ」はGit風完全再現ではなく「作業分岐」として扱う

#### 回答

スナップショットはコマ単位で保存し、日時で同期を取る。
指定の日時以前のところから新たに枝を生やすことも許容するし、あるコマだけ以前の状態に戻すことも許容する。
なるべく差分で保存したいが、セーブ・ロードの処理が重くなるようならフル保存にもできるようにしたい。
コマ内でのマージは許容しない。ページを構成する際に複数のスナップショットからコマを持ってくることは許容する。
コマ同士の重なりを許容する。原理的に競合はしないはず。

## 不足している要件

現状のメモには魅力的な機能が多い一方で、製品として必要な以下が不足している。

### 文書・データ要件

- 独自ファイル形式をどうするか
- 自動保存・クラッシュ復旧
- バージョン互換方針
- 作品内アセット管理

#### 回答

- ファイル形式の候補として、SQLiteベースの単一ファイル形式を検討する。理由は、部分ロード、索引管理、単一ファイルとしての扱いやすさに利点がありそうだから。
- ただし、タイル化ラスターの格納効率、差分スナップショットとの相性、破損時復旧のしやすさは別途検証が必要。
- 自動保存やクラッシュ復旧は後回し。
- バージョン互換のため、プロジェクトファイルには明示的なフォーマットバージョンを保持する。
- 作品内アセット管理はMVPでは最小限の参照と配置のみ対応し、高度な管理UIは専用プラグインとして後段で実装する。

なお、保存形式を決める前に、少なくとも以下の要件を満たす必要がある。

- コマ単位・ページ単位で部分ロードできる
- ラスターをタイル/チャンク単位で保存できる
- スナップショット差分保存またはフル保存を切り替えられる
- 将来的に圧縮・暗号化・破損復旧の余地を残せる

### 入出力

- 対応する入稿形式
  - PNG
  - PSD
  - PDF
  - TIFF
  - Web向け画像書き出し
- テキスト入稿形式
  - プレーンテキスト
  - Markdown
  - CSV/JSONベース脚本

#### 回答

入稿形式については、PNG, PSD, PDFを書き出しプラグインとして用意する。
テキストを流し込むための形式としては、MarkdownかJSONあたりをベースに、伏線マーカーやコマ区切り、重要セリフのハイライト、フォントやサイズの指定などを含んだ規格を整備し、専用テキストエディタをプラグインとして用意する。

MVPでは、まずMarkdownベースの人間可読な形式を第一候補とし、構造化情報が不足する場合にJSONベース形式を追加する。

### テキスト組版

- 縦書き/横書き
- ルビ
- 禁則処理
- 吹き出しとの連携
- フォント管理

#### 回答

MVPでは以下に限定する。

- 縦書き/横書きの切り替え
- 基本的な改行とテキストボックス配置
- 吹き出し内へのテキスト流し込み
- フォント、サイズ、太字程度の基本装飾

以下は後回しにする。

- ルビ
- 高度な禁則処理
- DTP級の細かい組版制御
- 自動文字詰めの高度最適化

### ペン・ブラシ

- ブラシエンジンの要件
- 手ブレ補正
- ベクター線対応の有無
- ラスタ/ベクター混在方針

#### 回答

まずはラスターをベースに開発して、そのうちベクターをベースにベクターであることを隠蔽するようなレイヤーを開発したい。

MVPでは、筆圧対応・基本的な手ブレ補正・不透明度/サイズ制御を備えたラスターブラシを対象とする。
ベクター線は後期実装とし、内部表現がベクターであることをUIから意識させないレイヤー設計を将来的に検討する。

### レイヤー・合成

- レイヤー種別
  - ラスタ
  - ベクター
  - テキスト
  - グループ
  - マスク
  - 調整レイヤー
- 合成モードの範囲
- 非破壊編集のポリシー

#### 回答

MVPでは、レイヤー種別はラスタ、テキスト、グループ、マスクに限定する。
ベクターと調整レイヤーは後回しにする。

合成モードは、通常、乗算、スクリーン、加算程度から始める。
非破壊編集はMVPでは最小限とし、まずはマスクと一部の変形履歴保持にとどめる。

### 操作性

- Undo/Redoの粒度と履歴長
- ショートカットカスタマイズ
- ペンタブ設定
- 複数ウィンドウ/複数モニタ対応
- アクセシビリティ

#### 回答

Undo/Redoはコマンド単位とし、対象コマまたは対象ページが明確にわかる履歴表示を目指す。
履歴長は固定値ではなくメモリ上限ベースで管理する。

ショートカットはMVPでもカスタマイズ可能にするが、まずは主要操作のみに限定する。
ペンタブは筆圧対応を優先し、詳細なデバイスチューニングは後回しにする。

複数ウィンドウ/複数モニタ対応は後期実装とし、MVPでは単一ウィンドウで破綻しない情報設計を優先する。
アクセシビリティは、キーボード操作・フォーカス移動・UIスケーリングに最低限対応する。

### 性能目標

最低でも目標指標がほしい。

例:

- 起動時間
- 100ページ作品を開く時間
- 4K相当キャンバスでのパン/ズームの目標FPS
- 書き出し時の最大メモリ使用量

#### 回答

MVP時点では以下を暫定目標とする。

- 起動時間: 開発機で3秒以内を目標
- 100ページ作品のサムネイル一覧表示: 初回5秒以内、再表示はより高速
- 4K相当キャンバスでのパン/ズーム: 体感上引っかからないことを優先し、60fps相当を目標
- バックグラウンド書き出し中も、主要UI操作が継続して行えること
- 書き出し時の最大メモリ使用量は、入力全体を無加工で丸ごと展開しない設計を前提に抑制する

数値目標は、最初の検証プロトタイプを実装したあとに再調整する。

## 技術選定の再結論

過去の検討では、既存のGUIフレームワークを土台にした構成も広く候補に含めていた。
しかし、要件を改めて整理すると、今回の必須条件は次の二つに集約される。

- ランタイムに Wasm または DSL のファイルを読み込んでパネルを表示できること
- そのパネルからアプリのホスト API を叩けること
- それと両立する形で、キャンバスを `wgpu` ネイティブで直接描画できること

この条件だけを見ると、一見そこまで極端には見えない。
ただし実際には、「UI が描ける」だけでは足りず、**UIランタイムの主導権も GPU 描画の主導権もホスト側が握る必要がある**ため、既存の単一フレームワークへ全面依存する構成は噛み合いにくい。

そのため、`altpaint` の技術選定は次の結論に更新する。

### 結論

- アプリケーションの描画フレームワークは内製する
- ウィンドウ/イベントループ/レンダリング基盤は `winit` + `wgpu` をベースにホスト側で直接管理する
- パネルUIは、Wasm/DSL をランタイムロードし、ホスト定義の中間表現として受け取って表示する
- パネルからの操作はホスト API または `Command` 経由で `app-core` に渡す
- JavaScript や Web フロントエンド技術は採用しない

### この結論に至った理由

要件ごとに見ると、以下のようになる。

#### 1. JavaScript を書きたくない

- WebView ベースやブラウザ前提のUIは採らない
- 拡張も JavaScript ベースにはしない

#### 2. Wasm/DSL の atomic コンポーネントをランタイムロードしたい

- コンパイル時に静的リンクされたUIだけでは不十分
- プラグインが「ホストが理解できるUI構造」を返せる必要がある
- 単なるスクリプト自動化ではなく、パネルとして常駐表示できる必要がある

#### 3. WGPU で直接描画したい

- キャンバスの render pass、GPU リソース、再描画タイミングをホストが直接持つ必要がある
- UI都合でキャンバス性能や更新粒度を制約されたくない

#### 4. 既存の単一フレームワークでは要件が素直に揃わない

- UI記述はできても、ランタイムロード型コンポーネントやホストAPIとの統合が弱い
- 逆にGPU描画はできても、パネルUIや拡張境界が弱い
- 結果として、各要件を無理やり一つの枠に押し込むより、責務を分離して自前で統合した方が合理的である

### 現実的なアーキテクチャ推論

以上を踏まえ、次の構成を基本とする。

- ベースレイヤー: `winit` + `wgpu`
  - スワップチェーンとフレームループはホストが直接管理する
- キャンバス描画
  - メインのレンダーパスはホスト実装とし、ズーム、パン、ブラシ、オーバーレイなどを `wgpu` ネイティブで描画する
- 動的UI統合レイヤー
  - ホスト内にパネルランタイムを持ち、DSL または Wasm から `PanelTree` のような中間表現を生成させる
  - パネルのレイアウト、ヒットテスト、描画、イベント配送はホストが行う
- プラグイン/API レイヤー
  - Wasm は `wasmtime` 等で実行し、ホスト関数を通じてアプリ API にアクセスする
  - フィルタ、入出力、テンプレート生成、バッチ処理、パネル表示をプラグインとして扱う

### 要するに難しいのか？

結論として、**条件そのものは不可能ではないが、既成の単一フレームワークに素直に乗る種類の要件ではない**。

難しさは「WGPU描画が難しい」というより、

- ランタイムロード型UI
- ホストAPIとの安全な接続
- ネイティブGPU描画

の三者を、主導権を失わずに同居させる点にある。

そのため本プロジェクトでは、描画フレームワークおよびUIランタイムを内製する方針を採る。

詳細は `docs/ARCHITECTURE.md` と `docs/RENDERING-ENGINE.md` で扱う。

## 推奨技術スタック

Rustを主言語とする前提で、現時点では以下がもっとも筋がよい。

### 結論

- 言語: Rust
- ウィンドウ/イベントループ: `winit`
- レンダリング: `wgpu`
- パネルUI: ホスト内製ランタイム
- パネル定義: DSL + Wasm の二段構え
- Wasmランタイム: `wasmtime`
- 描画/テクスチャ処理: GPU中心、CPUフォールバックあり
- 非同期実行: `tokio` または必要最小限の専用ワーカー
- 並列計算: `rayon`
- シリアライズ: `serde`
- ファイル形式: 独自プロジェクト形式（初期はJSON系暫定実装、将来的にタイル/チャンク指向へ移行）

この構成を推す理由は以下。

- ホストが GPU 所有権を直接握れる
- ランタイムロード型のパネルをホスト定義の中間表現へ正規化できる
- JavaScript や Web 技術を使わずに完結できる
- 標準パネルと外部パネルを同じ拡張モデルへ寄せやすい

ただし、内製ランタイムは当然コストが高いので、MVPでは UI コンポーネント種別を絞り、複雑なウィジェットや高度なレイアウトは後回しにする。

採用判断では少なくとも以下を見る。

- ホスト主導でフレームループを安定運用できるか
- パネル中間表現で標準UIを十分に記述できるか
- DSL / Wasm の両方から同じパネルモデルを返せるか
- キャンバス低遅延性をパネル系処理が阻害しないか

### レンダリング/画像処理

- GPU表示基盤は`wgpu`
- 画像デコードは`image`または対象形式ごとの専用crateを併用
- 色管理は初期段階では簡略化し、将来的にICC対応を検討
- CPU重処理は`rayon`で並列化
- 将来のGPUフィルタ実装を見据え、処理グラフを抽象化しておく

### 非同期・状態管理

- UIスレッドでは重い処理を直接行わない
- バックグラウンドジョブは`tokio`タスクまたは専用ワーカースレッドへ送る
- 作品状態はイベントソーシングまではやりすぎなので、まずは
  - 不変寄りのドキュメントモデル
  - コマンドベースUndo/Redo
  - バックグラウンドでサムネイルや書き出しを処理
    とする

### プラグイン

- ネイティブ動的ライブラリ読込より、初期はWASMを推奨
- 理由:
  - クラッシュ耐性
  - 配布容易性
  - セキュリティ境界
  - ABI地獄の回避
- 補足:
  - UIプラグインは Wasm が直接ネイティブUIやGPUを触るのではなく、ホストが理解できる `PanelTree` を返す方式を採る
  - DSL はより軽量な atomic パネルや設定UI向けに併用する

### データ保存

推奨案:

- プロジェクトはディレクトリベースまたは単一アーカイブ形式
- メタデータは`serde`で扱える構造化形式
- 大きいラスターはチャンク化・タイル化して保持
- スナップショットは差分中心で保存

## アーキテクチャ方針

### 層の分離

- `app-core`: ドキュメントモデル、履歴、コマ、テンプレート、書き出し
- `render`: キャンバス表示、タイル、GPUアップロード、キャッシュ
- `ui-runtime` / `ui-shell`: パネルレイアウト、描画、入力配送、プロファイル、ショートカット
- `plugin-host`: Wasmロード、権限管理、ホスト関数公開

この分離により、GUI技術を差し替えてもコアをできるだけ残せるようにする。

## まず作るべき検証プロトタイプ

技術選定を机上で終わらせないため、最初に以下の小プロトタイプを作る。

1. 100ページ相当のサムネイルとページ遷移が滑らかに動くか
2. 4Kキャンバスでパン/ズーム/描画が快適か
3. バックグラウンドPDF書き出し中にUIがどれだけ滑らかか
4. コマ中心ビューとページ中心ビューの切り替えが自然か
5. テキスト流し込みUIの認知負荷が低いか

## 次に詰めるべき論点

- 独自ファイル形式の設計
- コマ中心ワークフローのUIモック
- テンプレート/プロファイルのデータモデル
- テキスト組版機能の要件定義
- ブラシエンジンをどこまで自作するか

## 当面の結論

このアプリは「漫画向け機能があるペイントソフト」ではなく、**漫画制作の各工程を低認知負荷でつなぐ専用環境**として設計した方が筋がよい。

そのためには、最初から機能を盛りすぎず、

- コマ中心ビュー
- テキスト流し込み
- UIプロファイル
- 非同期で止まらない編集体験

をMVPの核に据えるのがよい。

技術スタックは、現時点では **Rust + `winit` + `wgpu` + ホスト内製UIランタイム + `tokio`/専用ワーカー + `rayon` + `wasmtime`** を第一案とし、コアとUIを疎結合にしてホスト側が主導権を持つ構成を採る。

ただし、内製UIランタイムの実装コストは高いため、最初のプロトタイプでは atomic なパネル部品に絞り、必要以上に広いUI自由度は最初から抱え込まない。

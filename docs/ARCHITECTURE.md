# altpaint アーキテクチャ設計

## この文書の目的

この文書は `docs/SKETCH.md` にある要件と判断を、実装責務・ランタイム境界・クレート構成まで落として定義するための中核設計文書である。

ここで明確にするのは主に以下である。

- 何を本体に持ち、何をランタイムロード対象にするか
- キャンバス描画とパネルUIをどう分離し、どう合成するか
- Wasm/DSL パネルがどのAPIまで叩けるか
- `app-core`、`render`、`storage`、プラグインホストの責務境界
- MVPで固定する点と、後から差し替える点

実装順序は `docs/ROADMAP.md`、描画系の詳細設計は `docs/RENDERING-ENGINE.md` を正とし、この文書では全体像と境界条件を定める。

## 方針転換の前提

2026-03-08 時点で、要求を改めて整理した結果、既存の単一GUIフレームワークへアプリ全体を委ねる方針は採らない。

理由は明確である。

- JavaScript を書かずに完結したい
- Web フロントエンド技術に依存したくない
- ランタイムに Wasm または DSL ファイルを読み込み、atomic なUIコンポーネントまたはパネルとして組み込める必要がある
- そのパネルからホストアプリの API を叩ける必要がある
- それとは独立に、キャンバスは `wgpu` を直接使ってネイティブ描画したい
- 既存候補はこの全条件を素直には満たさず、要件に対してフレームワーク都合が強すぎる

このため `altpaint` は、**描画フレームワークを内製し、UIランタイムもホスト主導で組み立てる**ことを基本方針とする。

既存ライブラリは部品として使ってよいが、アプリの責務分割や拡張モデルを外部フレームワークの前提に合わせて歪めない。

## 設計原則

`altpaint` では以下を優先する。

1. **GPU所有権をホストが握る**
   - ウィンドウ、サーフェス、スワップチェーン、主要 render pass はホストが直接管理する
   - キャンバス描画は `wgpu` によるネイティブパスを第一原則とする
2. **UIランタイムをホスト主導にする**
   - パネルはホストが定義した中間表現へロードし、描画・入力配送・API呼び出しはホストが管理する
   - 任意のGUIフレームワークに状態主導権を渡さない
3. **Wasm/DSL 拡張を最初から前提にする**
   - 標準パネルも、将来的に外部と同じモデルへ寄せられる形で作る
   - ただし MVP では権限とUI自由度を制限する
4. **キャンバスとパネルを分離する**
   - キャンバスは高頻度・低遅延の描画系
   - パネルは低頻度・構造化された UI 系
   - 更新頻度もレンダリングパイプラインも分ける
5. **コアはUI非依存で保つ**
   - `app-core` は GPU API や UI フレームワークの型に依存しない
   - UI はコマンドとクエリの境界を通じてのみ状態変更する
6. **標準機能も拡張可能な粒度で分ける**
   - ツールパレット、レイヤーパネル、履歴、書き出しUIなどは将来的に同じパネル基盤へ載せる

## システム全体像

大きく分けると、システムは次の6層で構成する。

1. `app-core`
   - ドキュメントモデル
   - コマンド適用
   - Undo/Redo
   - セッション状態
   - スナップ、選択、タスク要求の発行
2. `render`
   - キャンバス描画データの構築
   - タイル、キャッシュ、合成、オーバーレイ生成
   - GPUアップロード用の描画入力生成
3. `ui-runtime` 相当の層
   - パネル定義の読み込み
   - DSL/Wasm 実行
   - パネルツリーの中間表現化
   - レイアウト、ヒットテスト、UIイベント配送
4. `plugin-host`
   - Wasm モジュールロード
   - 権限管理
   - ホスト関数公開
   - パネル/入出力/フィルタ/自動化プラグインの起動
5. `storage`
   - プロジェクト読込/保存
   - タイル・スナップショット・メタデータ永続化
6. `apps/desktop`
   - `winit` ウィンドウ
   - `wgpu` 初期化とフレーム駆動
   - パネル合成
   - 入力収集
   - メインループとスケジューリング

## 単一フレームワークを採らない理由

今回の要件では、次の二つを同時に満たす必要がある。

- ランタイムにファイルからUI要素を読み込んで表示すること
- そのUIの上とは別に、WGPU ネイティブのキャンバス描画をホスト主導で成立させること

このとき重要なのは、「UIが描けること」ではなく、**UIのロード・描画・イベント・API呼び出しの全経路をホストが制御できること**である。

単一GUIフレームワークへ全面依存すると、以下のどれかが犠牲になりやすい。

- GPUリソースとフレームスケジューリングの所有権
- ランタイムロード型UIの自由度
- Wasm/DSL との相性
- UI拡張モデルの安定性

したがって、`altpaint` の立場は次の通りである。

- ウィンドウとフレームループは自前で持つ
- キャンバス描画は自前の `wgpu` パイプラインで持つ
- パネルUIはホストが定義した中間表現で持つ
- Wasm/DSL はその中間表現を返す側に寄せる
- 必要に応じて外部ライブラリは部品として使うが、アプリの中心規約はホストが決める

## 採用技術の現実解

現時点の現実解は以下である。

- ウィンドウ/イベントループ: `winit`
- GPU: `wgpu`
- Wasm ランタイム第一候補: `wasmtime`
- DSL: 独自の宣言的パネル DSL もしくは Rust で扱いやすいデータ記述形式
- パネル描画: 自前のUIランタイムで描画
- 2D テキスト/アイコン描画: 必要に応じて後から小さく採用

重要なのは、これらを「外から見たアプリの唯一の原理」にしないことだ。

特にパネルについては、Wasm が直接 `wgpu` やネイティブ view を触るのではなく、**ホストが理解できる `PanelNode` / `PanelTree` / `PanelAction` の中間表現を返す**方式を採る。

## 主要責務分離

### `app-core`

責務:

- `Document`、`Work`、`Page`、`Panel`、`LayerNode` の保持
- 作品の編集コマンド
- ツール状態、選択状態、ビュー状態
- Undo/Redo
- スナップショットモデル
- ホストAPIの意味論の定義

持たない責務:

- `wgpu` の型
- ウィンドウ型
- パネル描画の詳細
- Wasm 実行器の詳細

### `render`

責務:

- ドキュメント状態から描画対象を作る
- ラスタレイヤー合成
- タイルキャッシュ
- Dirty 範囲追跡
- オーバーレイ描画データ生成
- GPUへ渡す `RenderGraph` または `RenderFrame` の構築

持たない責務:

- ツールの意味論そのもの
- UIフレームワーク依存
- ストレージI/O

### `ui-runtime` / `plugin-host`

責務:

- パネルモジュールのライフサイクル管理
- Wasm/DSL のロード
- パネル中間表現の生成
- パネルレイアウト
- イベント配送
- ホストアクションの受け渡し

持たない責務:

- ドキュメント状態の直接改変
- GPUメインパスの運用
- プロジェクト保存形式の詳細

### `apps/desktop`

責務:

- ウィンドウ生成
- `wgpu` 初期化
- フレームスケジューリング
- キャンバス pass と UI pass の合成
- OS 入力をパネルランタイム/キャンバスへルーティング

## 依存方向

依存方向は原則として以下を守る。

- `apps/desktop` → `app-core`, `render`, `ui-shell`, `plugin-api`, `storage`
- `ui-shell` → `app-core`, `plugin-api`
- `render` → `app-core`
- `storage` → `app-core`
- `plugins/*` → `panel-sdk`

将来的に `plugin-host` クレートを追加する場合は以下とする。

- `plugin-host` → `plugin-api`, `app-core`
- `apps/desktop` → `plugin-host`

守るべきルール:

- `app-core` は UI 実装や GPU 実装に依存しない
- Wasm モジュールは `app-core` の内部型へ直接触れない
- パネルはホスト関数を介してのみ状態変更する
- 描画系クレートはプラグイン実行器の都合に引きずられない

## ランタイムUIモデル

### なぜ中間表現が必要か

ランタイムにロードした Wasm や DSL が、そのままネイティブUIウィジェットや GPU コマンドを発行すると、以下が破綻する。

- ABI 安定性
- セキュリティ・権限制御
- ホスト側の再レイアウト・再描画制御
- 将来的な UI 実装差し替え

ゆえにプラグインは、**UIを返す側**であって、**UI実装を所有する側**ではない。

### 基本データフロー

1. ホストがパネル定義ファイルを見つける
2. DSL ならパースする、Wasm ならインスタンス化する
3. `get_panel_tree()` 相当を呼び、`PanelTree` を得る
4. ホストがレイアウト・ヒットテスト・描画を行う
5. ユーザー入力を受けたらホストが `PanelEvent` を組み立てる
6. パネルへイベントを返し、必要なら `HostAction` を受け取る
7. `HostAction` を `Command` または非同期タスクへ変換する

### MVPで許可するUIノード

MVPでは、完全自由なUIツリーは許可しない。最初は atomic なパネル部品に絞る。

想定する最小ノード:

- `Column`
- `Row`
- `Section`
- `Text`
- `Button`
- `Toggle`
- `Separator`
- `List`
- `KeyValue`
- `IconButton`

MVPでは次を禁止または保留する。

- 任意シェーダでのパネル描画
- プラグイン側の独自GPUリソース管理
- 任意ネイティブウィジェット埋め込み
- 既存パネルツリー全体への自由パッチ

### Panel API の契約

パネルプラグインは概ね下記の契約を満たす。

- 入力
  - 読み取り専用の状態スナップショット
  - 自パネルローカル状態
  - ユーザーイベント
- 出力
  - `PanelTree`
  - `HostAction` の列
  - ローカル状態更新
- エラー
  - レンダリング不能
  - 権限不足
  - ホストAPI呼び出し失敗

成功条件は「ホストが安全に描画できる構造化UIを返すこと」であり、プラグインがホストの内部状態に直接到達できることではない。

## ホストAPI方針

パネルや自動化プラグインが叩くAPIは、直接内部構造を公開せず、意味のある操作単位に絞る。

例:

- `document.active_panel.get`
- `document.page.list`
- `tool.current.get`
- `tool.current.set`
- `layer.active.get`
- `layer.active.select`
- `command.dispatch`
- `export.request`
- `snapshot.create`
- `job.list`

### MVPの権限モデル

MVPでは権限は粗くてよいが、宣言は必須にする。

- `read.document`
- `read.selection`
- `read.jobs`
- `write.command`
- `write.export`
- `write.snapshot`

外部プラグインは manifest で権限申告し、ホストがロード時に検証する。

## キャンバスとパネルの合成

描画は少なくとも以下の順序で合成する。

1. 背景 clear
2. キャンバス下地
3. ドキュメント本体の描画結果
4. キャンバスオーバーレイ
   - 選択枠
   - スナップガイド
   - ブラシプレビュー
   - ページ/コマ境界
5. パネル背景
6. パネル内部ノード
7. ツールチップ、メニュー、モーダル

現行実装では、パネル内部ノードやホストの補助テキストは共有テキストレンダラで RGBA バッファへ直接描画する。フォントはまずシステムフォントを解決し、取得不能時のみビットマップフォールバックへ落とす。

ここで重要なのは、パネルがキャンバス描画パスを奪わないことと、逆にキャンバスがパネルレイアウト系の都合で不安定にならないことである。

当面は既存の workspace を土台にしつつ、次の方向へ整理する。

- `crates/app-core`
- `crates/render`
- `crates/storage`
- `crates/plugin-api`
- `plugins/*`
- `crates/ui-shell`
- `apps/desktop`

将来的に追加候補:

- `crates/plugin-host`
- `crates/panel-dsl`
- `crates/jobs`
- `crates/file-format`

### 各クレートの位置づけ

#### `crates/app-core`

- ドメインモデル
- コマンド
- セッション状態
- Undo/Redo
- スナップショット意味論

#### `crates/render`

- キャンバス内部表現から描画フレーム生成
- レンダリングエンジンの中核ロジック
- タイル/オーバーレイ/合成用の基礎型

#### `crates/ui-shell`

- いまは暫定的にパネルホスト表現を担う
- 将来的にはホスト非依存 UI 中間表現とレイアウトを担う層へ寄せる
- 特定GUIフレームワーク前提の責務は段階的に追い出す

#### `crates/plugin-api`

- パネル中間表現
- ホストアクション
- 状態スナップショット
- 権限宣言

#### `plugins/*`

- 標準パネルごとの独立フォルダ
- `.altp-panel` / Rust SDK ソース / 生成 Wasm の同居場所
- built-in と external を近づけるための実運用配置

#### `apps/desktop`

- デスクトップランタイム
- `winit` / `wgpu` のメインループ
- 実際のウィンドウ・入力・描画・合成責務

## 実装上の現実と移行方針

過去のコードベースには `Slint` を使った単一ウィンドウ構成が存在した。

これは要件検証用の過渡実装としては有効だったが、現在は撤廃済みである。

現時点での位置づけは以下である。

- 現在のホストは `winit` + `wgpu` を直接使う
- `wgpu` 直描画キャンバスは、そのまま将来構成へ接続できる中核である
- パネル中間表現の考え方は維持し、描画先もホスト実装で統一する

したがって、今後の移行では次を順に進める。

1. パネル表現をホスト定義の `PanelTree` へ固定する
2. `apps/desktop` 側に自前レイアウト/描画の最小パネルランタイムを維持拡張する
3. 入力・フォーカス・スクロール・ボタン押下をホスト管理で拡張する
4. その上で DSL/Wasm ローダを接続する

## データモデルの骨格

内部モデルの骨格は維持する。

- `Work`
  - `Page`
    - `Panel`
      - `LayerNode`

これは漫画制作の都合上、ページ概念が最終出力で依然重要だからである。

ただしUIはコマ中心に組む。

- 編集対象の主軸はコマ
- 履歴表示もコマ/ページ文脈を持つ
- スナップショットはコマ単位を中心に据える

## 非目標と制約

MVPでは次をやらない。

- Web技術ベースのUI
- JavaScript による拡張
- プラグインからの任意GPU描画
- パネルツリー全体への自由差し替え
- 高度なマルチウィンドウ運用

また次を前提にする。

- Wasm は UI の構造とホスト操作要求を返すために使う
- パネル描画そのものはホストが行う
- キャンバスは `wgpu` ネイティブ描画を最重要要件とする

## 設計上の主要リスク

### 1. UI ランタイム内製コスト

独自レイアウト、ヒットテスト、フォーカス、テキスト描画を持つ必要がある。

対策:

- MVPではコンポーネントを限定する
- 高度なウィジェットは後回しにする
- 標準パネルを少数で回しながら API を固める

### 2. Wasm ABI の安定化

対策:

- 直接ポインタ共有を避ける
- C ABI かシリアライズ済みバッファで受け渡す
- API は coarse-grained に保つ

### 3. レンダリングエンジン肥大化

対策:

- `Document` の意味論と描画パイプラインを分離する
- タイル・合成・オーバーレイ・提示の層を明確に分ける
- 詳細は `docs/RENDERING-ENGINE.md` に切り出して統制する

## この文書の結論

`altpaint` は、既存GUIフレームワークにアプリの中心責務を委ねるのではなく、次の構造で要件を満たす。

- ホストが `winit` + `wgpu` を直接管理する
- キャンバスはネイティブWGPU描画で実装する
- パネルは Wasm/DSL からランタイムロードする
- パネルはホスト定義の中間表現を返し、ホストが描画する
- パネルからの操作はホストAPI経由で `app-core` に渡す

これにより、「ランタイムロード可能なUI拡張」と「ネイティブGPU描画キャンバス」を、同じホストの支配下で両立させる。

#### `Page`

主な責務:

- ページサイズ
- 見開き情報
- 断ち切り、ノド、小口などの基準保持
- コマ配置の親コンテナ

主なフィールド例:

- `page_id`
- `page_index`
- `dimensions`
- `bleed`
- `safe_area`
- `panel_ids: Vec<PanelId>`
- `page_snapshot_refs`

#### `Panel`

主な責務:

- コマ境界
- コマ作業単位としての識別
- レイヤーツリーのルート
- テキスト流し込み対象
- スナップショット粒度

主なフィールド例:

- `panel_id`
- `page_id`
- `frame_geometry`
- `overlay_geometry`
- `layer_root_id`
- `text_flow_ref`
- `snapshot_head`

`overlay_geometry` は、コマ外へ飛び出す描画やマスク除外グループの扱いのために持たせる。

#### `LayerNode`

初期対応するレイヤー種別:

- `RasterLayer`
- `TextLayer`
- `GroupLayer`
- `MaskLayer`

後期対応候補:

- `VectorLayer`
- `AdjustmentLayer`

すべてのレイヤーは木構造上に置かれ、以下の共通属性を持つ。

- `id`
- `name`
- `visible`
- `locked`
- `opacity`
- `blend_mode`
- `children`（グループ系のみ）

## UI状態とドキュメント状態の分離

状態は大きく 3 つに分ける。

### 1. 永続化される作品状態

例:

- ページ構造
- コマ構造
- レイヤー内容
- テンプレート参照
- スナップショット

### 2. セッション状態

例:

- 開いているパネル
- 選択中のコマ
- カメラ位置
- 展開中のツリー
- 実行中ジョブの進捗

このセッション状態は、特定UI実装のローカルウィジェット状態へ閉じ込めず、ホスト側が保持する明示的な状態として管理する。

特に以下はUIフレームワーク内部状態に沈めない。

- ドック配置
- アクティブパネル
- フォーカス対象
- キャンバスとパネル間の入力モード
- ショートカットの現在有効なルーティング先

これは基本的に作品データとは分ける。

### 3. 一時的UI状態

例:

- ホバー状態
- ドラッグ中のガイド
- 一時プレビュー
- IME変換中テキスト

UIランタイム内部の一時キャッシュや一時ノード状態も原則としてここに属する。

これは保存しない。

## キャンバスとレンダリング設計

### 表示パイプライン

描画は大きく次の段階で行う。

1. ドキュメント状態から表示対象を抽出
2. 表示領域に必要なタイルを解決
3. GPU/CPU キャッシュを参照
4. ラスター、テキスト、オーバーレイを合成
5. UIフレームへ出力

### タイル戦略

MVPではラスター中心のため、キャンバス全体を毎回フル再構築しない。

方針:

- ラスターはタイル単位で保存/ロード
- 表示中のビューポート周辺を優先ロード
- よく見る倍率のキャッシュを持つ
- 大きい画像は遅延デコードする

### オーバーレイ

オーバーレイは本描画とは別レイヤーで持つ。

対象例:

- コマ境界
- スナップ線
- 距離表示
- 選択範囲
- 現在ツールのプレビュー
- 前後ページ/前後コマの簡易表示

これにより、本体のラスター描画を毎回壊さずUI反応を保つ。

## 入力処理

入力は「道具」と「対象」の組み合わせで考える。

- 道具
  - ブラシ
  - 消しゴム
  - コマ分割/変形
  - テキスト流し込み
  - 移動
  - スナップ編集
- 対象
  - ページ
  - コマ
  - レイヤー
  - テキストオブジェクト

入力ルーティング方針:

1. 現在ツールを判定
2. フォーカス対象を判定
3. 必要ならスナップ候補を算出
4. プレビューを表示
5. コミット時に `Command` を発行

直接ミューテーションするのではなく、最終的にはできるだけ `Command` に落とす。

## Undo/Redo とコマンドモデル

Undo/Redo はコマンドベースで実装する。

### 理由

- UIフレームワークから独立して扱える
- スナップショットとの相性がよい
- プラグイン操作も同じ履歴へ流し込みやすい

### 代表的なコマンド

- `CreatePage`
- `SplitPanel`
- `MovePanelBoundary`
- `ReorderPanel`
- `InsertLayer`
- `RenameLayer`
- `ApplyBrushStroke`
- `InsertTextFlow`
- `TransformSelection`
- `RestorePanelSnapshot`

### 履歴ポリシー

- 履歴はメモリ上限ベースで管理
- 長いストロークは内部的に圧縮可能にする
- UIでは「何に対する操作か」が見える履歴名を表示する

## スナップショットと分岐モデル

このアプリの独自性として、履歴とスナップショットは「コマ単位」を主軸にする。

### 基本方針

- スナップショットはコマ単位で保存
- 日時ベースで並ぶ
- 過去時点から新しい枝を生やせる
- 特定コマだけを以前の状態に戻せる
- コマ内部の自動マージは行わない
- ページ再構成時に異なる時点のコマを組み合わせられる

### モデル案

- `PanelSnapshot`
  - `snapshot_id`
  - `panel_id`
  - `parent_snapshot_id`
  - `created_at`
  - `storage_ref`
  - `save_strategy` (`delta` / `full`)

### 保存戦略

- 基本は差分保存を優先
- 一定間隔または条件でフル保存を挟む
- ロードコストが悪化したら自動的に再パック可能にする

## スナップ設計

スナップ対象は少なくとも以下。

- ページ枠
- 断ち切り線
- ノド/小口基準
- コマ枠
- 隣接コマの辺
- 余白ガイド
- セーフエリア

MVPでは以下を実装対象とする。

- 近傍候補の自動提示
- 拘束方向の簡易表示
- 距離の簡易表示

後期実装候補:

- CADライクな精密数値入力
- スナップ候補の優先度カスタマイズ
- 高度な視覚フィードバック

## テキスト流し込みアーキテクチャ

テキスト流し込みはこのアプリの核機能の一つとして扱う。

### 目的

- テキストからネームへの遷移を速くする
- コマごとの情報量を先に把握する
- セリフ構造をページレイアウトへ反映しやすくする

### 構成要素

- `TextSource`
  - Markdown または JSON 入力
- `TextParseResult`
  - セリフ、地の文、マーカー、強調などの構造化結果
- `TextFlowPlan`
  - どのコマへどのテキストを流すか
- `TextLayout`
  - 実際のボックス配置結果

### MVP方針

- 縦書き/横書きに対応
- 吹き出しまたはテキストボックスへの流し込み対応
- 基本装飾のみ対応
- ルビや高度禁則は後回し

## プロファイルとパネル構成

制作プロファイルは UI の見た目ではなく、**作業文脈の切り替え**として扱う。

### プロファイルに含むもの

- 表示するパネル種類
- パネル配置
- ツールパレット構成
- ショートカットの一部
- 補助表示のON/OFF
- デフォルト作成レイヤー名

例:

- ネーム用
- ラフ用
- 線画用
- 仕上げ用

### パネルの種類

MVPで持つ候補:

- キャンバス
- ページ一覧
- コマ一覧
- レイヤー
- ツール
- テキスト流し込み
- テンプレート
- スナップショット履歴
- ジョブ進捗

このうち、キャンバスは基盤機能として本体が直接持つ。
それ以外のパネルは、原則として標準プラグインとして実装する。

つまりMVPの見た目としては標準機能に見えても、内部実装としては以下に分かれる。

- 基盤機能
  - キャンバス
  - パネルホスト
  - ショートカットルーティング
- 内部標準プラグイン
  - ページ一覧
  - コマ一覧
  - レイヤー
  - ツール
  - 色
  - テキスト流し込み
  - テンプレート
  - スナップショット履歴
  - ジョブ進捗

これにより、標準機能と将来の拡張機能を同じパネルモデルの上で扱えるようにする。

### パネル構成方針

ホストUIランタイム上では、各パネルを独立した宣言的ノードツリーとして構築する。

方針:

- 各パネルは小さなviewの合成で組む
- 可能な限り読み取り専用状態とアクション送信を分ける
- パネル間共有状態は `ui-shell` 側で束ねる
- 作品状態変更は `Command` またはホストメッセージへ落とす

これにより、標準パネルを atomic に保ったまま、後からプロファイル単位の再配置や差し替えを行いやすくする。

## プラグインアーキテクチャ

### 方式

- アーキテクチャ上、標準UIも外部拡張も「プラグイン」として扱う
- 外部配布プラグインの実行形態としてはWASMを第一候補とする
- `wasmtime` 上で実行する
- ネイティブ ABI はMVPでは公開しない

ここでいうプラグインは二層に分かれる。

- 内部プラグイン
  - 本体同梱の標準機能
  - 同一workspace内のRustモジュールとして実装可能
  - パネルUIの大半はこれで構成する
- 外部プラグイン
  - 公開API越しに追加される拡張
  - WASMを第一候補とする

両者は可能な限り同じ概念モデルを共有するが、MVPでは内部プラグインの方がやや広いAPIを使ってよい。

### MVPで許すプラグイン種別

- 内部標準パネル
- 新パネル追加
- 入出力
- フィルタ
- テンプレート生成
- バッチ処理
- ブラシ/形式コンバータ

MVPでは、ツールパレット、色パレット、レイヤー編集、テキスト流し込み、テンプレート管理、スナップショット履歴などを内部標準プラグインとして実装する想定とする。

### 後期に検討する拡張

- 既存パネルへの拡張ポイント注入
- パネル差し替え
- UIパッチ
- 自動化スクリプト

### 権限モデル

最低限、以下の権限を分ける。

- `read_document`
- `write_document`
- `read_assets`
- `write_assets`
- `register_panel`
- `export_file`
- `import_file`
- `run_batch_job`

プラグインは宣言した権限だけを要求し、ホスト側で承認する。

内部プラグインは本体同梱であるため別の信頼モデルを持てるが、設計上は外部プラグインと同じ権限体系をできるだけ共有する。

### ホストAPIの原則

- ドキュメント構造の生ポインタを渡さない
- 直接UIツリーを書き換えさせない
- コマンドベースまたはメッセージベースで作用させる
- 失敗しうる操作は必ずエラー境界を設ける

## 保存形式

保存形式はまだ確定しないが、現時点では以下の要件を満たす必要がある。

- ページ/コマ単位の部分ロード
- タイル/チャンク単位のラスター保存
- スナップショットの差分/フル保存切り替え
- 将来の圧縮や破損復旧に対応できること
- バージョン情報を明示的に持てること

### 候補A: SQLiteベース単一ファイル

利点:

- 単一ファイルで扱いやすい
- 索引管理しやすい
- 部分読込の設計次第でまとまりがよい

懸念:

- 巨大バイナリの格納戦略を慎重に設計する必要がある
- 差分保存との相性は検証が必要

### 候補B: ディレクトリベース形式

利点:

- タイル/チャンク管理を素直に表現しやすい
- デバッグしやすい
- 破損時の切り分けがしやすい

懸念:

- ファイル数が増えやすい
- 配布や移動の利便性が落ちる

### 暫定方針

- MVP前に保存形式検証プロトタイプを作る
- UI実装より前に、差分スナップショットとタイル保存の相性を確認する

## 非同期ジョブシステム

重い処理は `jobs` 層へ送る。

対象例:

- PDF書き出し
- PSD/PNG変換
- サムネイル生成
- 大きなフィルタ処理
- タイル再圧縮
- プラグインによるバッチ処理

### 優先度

最低でも3段階を持つ。

- 高: ユーザー操作直後に必要な処理
- 中: 表示改善のための先読み
- 低: 書き出し、再圧縮、バックグラウンド整備

### キャンセル

- 書き出しや再生成は取消可能にする
- ストローク中に無効になった古いプレビュー生成は破棄する

## エラー処理と破損耐性

MVPでも最低限以下を守る。

- 読めない一部データがあっても作品全体を即死させない
- プラグイン失敗はホスト本体に波及させない
- ストレージ破損時は可能な範囲で読み出す
- 長時間処理は進捗と失敗理由を出す

## MVPの成功条件

最低限、以下を満たせればMVPとして意味がある。

- コマ中心ビューで主要工程が迷わず進められる
- テキスト流し込みからコマ編集までの導線が自然である
- 100ページ規模の作品で一覧や遷移が破綻しない
- 4K相当キャンバスでパン/ズーム/描画が実用的に快適である
- PDF書き出し中も主要UIが止まらない
- プラグインパネルを1つ以上追加できる

## 当面の結論

`altpaint` のアーキテクチャは、**コマ中心ワークフローを支えるコア**と、**重い処理でも止まらないUI/描画基盤**を中心に組み立てる。

そのために当面は、

- UIはホスト主導のランタイムとして実装し、特定GUIフレームワークへ中心責務を委ねない
- UIパネルはホスト定義の中間表現で扱い、プラグイン境界では特定UI実装の固有型に閉じないようにする
- キャンバスは `winit` + `wgpu` を前提に低レイヤーで実装し、パネル系ランタイムとは独立に保つ
- 描画は `wgpu` を前提にタイル/キャッシュ設計を行う
- 保存はコマ単位スナップショットを中心に設計する
- プラグインはWASMで安全側に倒す
- 機能追加より先に、キャンバス、保存、UI導線の3つを検証する

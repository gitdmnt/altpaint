# Panel Performance TODO

## 背景

キャンバス描画は dirty rect を使って部分更新できる一方、パネル描画は CPU 側の再レイアウト・文字描画・ビューポート切り出しの比率が高く、入力時やスクロール時に FPS が落ちやすい。

## 観測している主な問題

1. パネル内容の再構築コストが高い
   - `UiShell::build_panel_content_surface()` で縦に長いサーフェス全体を再生成する
   - テキストやノード数が多いほど再描画コストが増える

2. スクロール時にもビューポート切り出しコピーが発生する
   - `viewport_panel_surface()` が可視行を毎回新しいサーフェスへコピーする
   - GPU スクロールではなく CPU メモリコピー主体になっている

3. 文字描画が CPU ボトルネックになりやすい
   - パネルはラベル・ボタン・セクション見出しなどでテキスト量が多い
   - wrap / measure / draw がノード数ぶん繰り返される

4. レイアウト計測と描画が分離されておらず二重コストになりやすい
   - `measure_panel_tree()` / `measure_node()` と `draw_panel_tree()` が類似走査を行う
   - ノード木が大きいほど CPU 時間が伸びる

5. キャンバス経路との差が大きい
   - キャンバスは dirty rect と部分 upload が前提
   - パネルは基本的にサイドバー全体の再合成に寄りやすい

## 2026-03-09 時点の対処

- スクロール量を 1 行高さ基準から固定 48px 相当に引き上げ、体感速度を改善した
- パネル内容キャッシュ自体は維持し、スクロールで全 UI 更新を走らせないようにしている

## 次の改善候補

- 可視領域単位のパネルタイル化
- テキスト計測結果のキャッシュ
- ノードごとのレイアウト結果キャッシュ
- スクロール時の差分 blit / オフセット再利用
- パネル dirty rect の導入
- 頻繁に変わらないパネルの静的サーフェス化
